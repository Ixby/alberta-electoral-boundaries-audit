# Proposed Project Directory Reorganization

The arrival of the official shapefiles fundamentally changes the nature of the repository. We are no longer a "geometry construction" project; we are a "forensic analysis" project. The current folder structure carries a lot of technical debt from the blocked phase, cluttering the root directory and mixing active analytical code with deprecated validation pipelines.

Here is a proposed cleanup and reorganization to make the repository "Pitch Ready" for external peer reviewers or legal counsel.

## 1. The Root Directory Purge
Currently, the root directory is cluttered with generated reports, images, and one-off scripts. 

**Action:** Move all final deliverables to a dedicated `outputs/` directory.
- `report_academic.md` / `.pdf` ➔ `outputs/academic_report/`
- `report_public.md` / `.pdf` ➔ `outputs/public_report/`
- `presentation_dr_bratt.pptx` ➔ `outputs/presentations/`
- `infographic.png` / `infographic_v2.png` ➔ `outputs/assets/`

**Action:** Consolidate root-level runner scripts.
- Keep `run_audit.py` (the master pipeline) at the root.
- Move `build_report.py`, `run_master_qa.py`, `generate_ppt.py`, and `add_images.py` into `analysis/scripts/automation/`.

## 2. Archiving the "Blocked Phase" (DPG & Validation)
Since all Derived Provisional Geometries (DPGs) and their validation constructs (like v11) are officially deprecated, they should be fully isolated from the active codebase so reviewers don't accidentally review dead code.

**Action:** Consolidate everything into `historical/`.
- Move the `dpg_validation/` directory into `historical/dpg_validation/`.
- Move the newly created `analysis/scripts/dpg_archive/` into `historical/dpg_scripts/`.
- Move `pipeline_snapshots/` into `historical/pipeline_snapshots/`.
- Move any `data/shapefiles/derived/v0_*.gpkg` files into `historical/dpg_shapefiles/`.

*Rationale:* This explicitly signals that these files are retained for methodological provenance only and are not part of the active forensic audit.

## 3. Data Directory Strict Typing
The `data/` folder likely contains a mix of official inputs, intermediate outputs, and old generated files.

**Action:** Enforce a strict Bronze/Silver/Gold data architecture.
- `data/raw/` (Bronze): Immutable inputs exactly as downloaded (e.g., `alberta_2026_official_shapefiles.zip`, Census 2021 raw CSVs).
- `data/processed/` (Silver): Cleaned substrates ready for analysis (e.g., `va_polygons_with_full_2023_votes.gpkg`).
- `data/outputs/` (Gold): The final `.csv` or `.json` files generated by the MCMC ensemble, compactness scores, and packing/cracking analyses that feed directly into the reports.

## 4. Consolidating "Private" Context
There are currently two private folders (`private/` and `private_notes/`) which likely contain emails, API keys, or sensitive unredacted notes.

**Action:** Merge into a single `.gitignore`'d directory.
- Merge `private/` and `private_notes/` into `private_workspace/`.
- Ensure `private_workspace/` is strictly documented in `.gitignore` to prevent accidental commits of sensitive PO correspondence or unredacted EA contacts.

## 5. Refactoring Plan: Config-Driven Portability
Currently, the analytical scripts rely heavily on hardcoded paths (`DATA = ROOT / "data"`) and hardcoded shapefile references (`v0_9_majority`, etc.). This tightly couples the logic to the exact state of this repository. To ensure the scripts can be extracted, sent to peer reviewers, and run on any machine against arbitrary maps, a comprehensive refactoring pass will be executed alongside the file moves:

**Action 1: Eliminate Absolute/Hardcoded Paths**
- All scripts will use `pathlib` resolving relative to the script's execution location, or strictly respect paths passed in at runtime. Absolute paths will be completely removed.

**Action 2: Introduce a Central Configuration File**
- A `config.yaml` (or `.json`) will be introduced at the root directory.
- Rather than hardcoding shapefile names or data paths inside scripts, the execution orchestrator (`run_audit.py` or similar) will read the config file.
- The config file will dictate:
  - The path to the demographic substrate (e.g., Voting Areas).
  - The path(s) to the map(s) being evaluated (Majority, Minority, Enacted).
  - Expected column names for votes (e.g., `va_ndp`, `va_ucp`), populations, and geographic IDs.
  - Output directories.

**Action 3: Make Core Logic Agnostic**
- Scripts like `packing_cracking_analysis.py` and `mcmc_ensemble.py` will be modified to accept `GeoDataFrame` objects and metric columns as generic arguments, rather than hardcoding "NDP" vs "UCP" or "majority" vs "minority".
- Any remaining hardcoded values (such as an exact `total_population` anchor, if required mathematically and not dynamically computable) will be explicitly documented and justified within the script or config file.

## 6. Technical Debt & Proposed Resolutions

While preparing for portability, I've identified several areas of deep-seated technical debt in the active pipeline that must be resolved:

**Debt 1: Copy-Paste Script Proliferation**
- *Issue:* We have four different MCMC runner scripts (`mcmc_ensemble.py`, `mcmc_ensemble_250k.py`, `mcmc_ensemble_canonical.py`, `mcmc_full_coverage_rescore_v2.py`). Whenever logic needed changing, the script was duplicated rather than parameterized.
- *Resolution:* Consolidate into a single `mcmc_runner.py` that accepts `--steps`, `--maps`, and `--output` flags via the new config.

**Debt 2: Redundant Automation Tools**
- *Issue:* Multiple report generators (`build_report.py`, `build_pdf.py`, `build_academic_pdf.py`).
- *Resolution:* Merge into a single `automation/build_reports.py` with specific target flags (e.g., `--academic`, `--public`).

**Debt 3: No Centralized Data Loader**
- *Issue:* All ~60 analytical scripts independently invoke `gpd.read_file()`. If the CRS mapping fails, or columns need renaming, it breaks everywhere.
- *Resolution:* Create `analysis/scripts/utils/data_loader.py` to handle all shapefile I/O, CRS standardization, and structural validation centrally.
